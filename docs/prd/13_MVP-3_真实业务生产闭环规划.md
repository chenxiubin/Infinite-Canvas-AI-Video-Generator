## 1. MVP-3 目标

### 1.1 一句话定义

把平台从"测试闭环"推进到"台历/挂历 AI 视频批量生产闭环"——用户上传素材包 → 选择模板 → 批量生成 → 审核修正 → 合成导出，全流程可操作、状态可追踪、失败可恢复。

### 1.2 最小闭环

```
产品素材包 → 视频分镜模板 → 批量产品链 → 图生视频节点 → 状态管理 → 失败重跑 → 合成导出
```

### 1.3 量化目标

| 指标 | 目标值 |
|------|--------|
| 支持产品线 | 2 条（台历 + 挂历） |
| 每条产品链的分镜节点数 | 6 个 |
| 单节点视频时长范围 | 3～6 秒 |
| 单条成片总时长 | 25～30 秒 |
| 单批次支持产品数 | 1～50 条 |
| 单节点生成成功率（Mock） | ≥ 85% |
| 失败节点恢复 | 单节点重跑，不影响其他节点/产品链 |
| 批次状态 | queued → running → completed / partially_completed / failed |

### 1.4 MVP-3 与 MVP-2 的关系

MVP-2 已经实现了批量克隆、节点状态机、失败重跑、合成触发、批次追踪。MVP-3 在此基础上：

- **保留**：所有 MVP-2 的状态机、批量机制、安全校验
- **新增**：真实素材资产包、升级版分镜模板、统一模型网关、审核体系、导出闭环
- **替换**：占位 mock 视频 → 统一模型网关接口（先 mock，接口设计为真实可替换）
- **不改变**：画布交互模型、React Flow 渲染、Zustand 状态管理

## 2. 产品流程

### 2.1 端到端流程

```
Step 1: 用户上传产品素材包
  ├── 台历素材包：main + detail_1 + detail_2 + scene + motion + brand (6 张)
  └── 挂历素材包：main + detail_1 + detail_2 + scene + motion + brand (6 张)

Step 2: 选择产品线与视频模板
  ├── 台历模板 (tpl_desk_v3)
  └── 挂历模板 (tpl_hanging_v3)

Step 3: 批量创建产品实例
  ├── 按 SKU 分组素材，匹配模板
  ├── 每个 SKU 生成一条产品链 (ProductInstance)
  ├── 每条链 6 个 InstanceNode
  └── 缺失素材的角色标记为 pending，阻止该节点生成

Step 4: 提交批量生成任务
  ├── 创建 BatchTask
  ├── 对每条链的每个节点调用 model_gateway
  ├── 节点独立状态 tracked：pending → generating → success / failed
  └── 全部节点成功后自动触发 merge → completed

Step 5: 审核修正
  ├── 预览每个节点的生成结果（视频/封面）
  ├── 标记审核状态：pending / approved / rejected
  ├── 失败/驳回节点 → 修正提示词/替换素材 → 重跑
  └── 依赖节点链：S01_main 生成后才生成 S02

Step 6: 合成导出
  ├── 单链所有节点 approved → 触发合成
  ├── 合成后输出完整视频 + 平台规格适配
  └── 下载 / 查看导出记录
```

### 2.2 产品素材包结构

```
素材包（按 SKU 组织）：
  SKU2027-A01/
    ├── SKU2027-A01_main.jpg          # 产品主图
    ├── SKU2027-A01_detail1.jpg       # 纸张/印刷细节
    ├── SKU2027-A01_detail2.jpg       # 结构/装订细节
    ├── SKU2027-A01_scene.jpg         # 使用场景
    ├── SKU2027-A01_motion.jpg        # 运镜参考图（可选，用于视频首帧）
    └── SKU2027-A01_brand.jpg         # 品牌尾帧（可选，有默认值）
```

### 3.2 需要新增/扩展的数据模型

#### 3.2.1 Product 产品线表（新增）

当前产品线信息散落在 templates 表和前端硬编码中，缺乏独立实体。

```sql
CREATE TABLE products (
    id TEXT PRIMARY KEY,                      -- 'desk_calendar' | 'wall_calendar'
    name TEXT NOT NULL,                        -- '台历' | '挂历'
    display_name TEXT NOT NULL,                -- '新年台历'
    default_template_id TEXT,                  -- 默认模板
    description TEXT,
    created_at REAL NOT NULL,
    updated_at REAL NOT NULL
);
```

**影响旧数据**：否。现有 `templates.product_id` 作为外键关联。

#### 3.2.2 ProductAsset 产品素材资产（新增）

独立于画布绑定的素材资产表，支持批量上传和按 role 管理。

```sql
CREATE TABLE product_assets (
    id TEXT PRIMARY KEY,
    product_sku TEXT NOT NULL,                 -- 所属产品 SKU
    product_id TEXT NOT NULL,                  -- 所属产品线
    role_key TEXT NOT NULL,                    -- main / detail_1 / detail_2 / scene / motion / brand
    role_name TEXT,                            -- 主图-正面 / 细节-纸张质感 等
    filename TEXT NOT NULL,
    original_filename TEXT NOT NULL,
    url TEXT NOT NULL,
    file_size INTEGER,
    width INTEGER,
    height INTEGER,
    format TEXT,                               -- jpg / png / webp
    uploaded_at REAL NOT NULL,
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

**影响旧数据**：否。与现有 `assets` 表并存，`assets` 表继续服务于单链场景。

#### 3.2.3 GenerationJob 生成任务（替代现有 `jobs` 表的扩展）

当前 `jobs` 表过于简化（仅 type + target_id + status + result_json），需要结构化记录每次模型调用的完整信息。

```sql
CREATE TABLE generation_jobs (
    id TEXT PRIMARY KEY,
    instance_node_id TEXT NOT NULL,             -- 目标节点
    instance_id TEXT NOT NULL,                  -- 所属产品链
    batch_id TEXT,                              -- 所属批次（可选）
    shot_key TEXT NOT NULL,                     -- S01_main 等
    product_type TEXT NOT NULL,                 -- desk_calendar / wall_calendar
    model_name TEXT NOT NULL,                   -- kling / jimeng / runway / mock
    model_version TEXT,                         -- 模型版本号
    input_image_url TEXT NOT NULL,              -- 输入图片
    prompt_text TEXT NOT NULL,                  -- 完整提示词
    prompt_version TEXT,                        -- 提示词模板版本
    duration_requested REAL,                    -- 请求时长
    duration_actual REAL,                       -- 实际生成时长
    aspect_ratio TEXT,                          -- 9:16 / 16:9 / 1:1
    seed INTEGER,                               -- 随机种子
    text_lock_enabled INTEGER DEFAULT 0,        -- 文字保护
    status TEXT NOT NULL DEFAULT 'queued',      -- queued / processing / success / failed
    video_url TEXT,                             -- 输出视频
    cover_url TEXT,                             -- 封面图
    error_code TEXT,                            -- 结构化错误码
    error_message TEXT,                         -- 人类可读错误信息
    retry_count INTEGER DEFAULT 0,              -- 重试次数
    cost_estimate REAL,                         -- 预估费用
    created_at REAL NOT NULL,
    started_at REAL,
    completed_at REAL,
    FOREIGN KEY (instance_node_id) REFERENCES instance_nodes(id),
    FOREIGN KEY (instance_id) REFERENCES instances(id)
);
```

**影响旧数据**：否。现有 `jobs` 表保留，`generation_jobs` 为新增表。

#### 3.2.4 ReviewRecord 审核记录（新增）

```sql
CREATE TABLE review_records (
    id TEXT PRIMARY KEY,
    generation_job_id TEXT NOT NULL,
    instance_node_id TEXT NOT NULL,
    reviewer TEXT,                              -- 审核人
    video_duration_ok INTEGER,                  -- 时长合格
    product_shape_ok INTEGER,                   -- 产品形状正常
    text_readable_ok INTEGER,                   -- 文字可读
    no_obvious_distortion INTEGER,              -- 无明显变形
    no_black_frame INTEGER,                     -- 无黑帧
    no_watermark INTEGER,                       -- 无水印
    review_status TEXT NOT NULL DEFAULT 'pending',  -- pending / approved / rejected
    review_notes TEXT,                          -- 审核备注
    created_at REAL NOT NULL,
    updated_at REAL,
    FOREIGN KEY (generation_job_id) REFERENCES generation_jobs(id),
    FOREIGN KEY (instance_node_id) REFERENCES instance_nodes(id)

```

**影响旧数据**：否。全新表。

### 3.3 现有 instance_nodes 需要新增的字段

| 字段 | 类型 | 用途 | 默认值 |
|------|------|------|--------|
| `prompt_text` | TEXT | 该节点生成时使用的完整提示词 | NULL |

### 3.4 关键设计约束

- **画布坐标 (x, y) 仅用于 UI 渲染**，不能作为业务生成逻辑的依据
- **业务逻辑依赖结构化字段**：product_type, asset_role, shot_key, instance_id, batch_id
- **模板节点不保存素材 URL**，克隆时才产生具体绑定
- **批量操作的计数类字段使用原子操作**（`UPDATE ... SET count = count + 1`），不是"先读后写"

### 4.3 图生视频提示词结构

每个分镜节点的提示词由结构化字段拼接而成。**关键约束**：产品图案、文字、日期、印刷内容不能变形、不能乱码、不能重绘。

```json
{
  "shot_key": "S01_main",
  "prompt_template": {
    "base": "产品展示视频，{product_type}，{shot_name}",
    "composition": "{shot_size}，{camera_move}，居中构图",
    "lighting": "{lighting_mood}",
    "quality": "高清晰度，产品形状稳定，文字清晰可读，无变形无乱码",
    "negative": "模糊，变形，扭曲，文字乱码，图案重绘，产品形状改变，黑帧，水印",
    "protect_text": true,
    "protect_product_shape": true
  }
}
```

**拼接后示例**（S01_main, 台历）：

> 产品展示视频，台历，主视觉展示。中景，缓慢推进，居中构图。暖光节日氛围。高清晰度，产品形状稳定，文字清晰可读，无变形无乱码。Negative: 模糊，变形，扭曲，文字乱码，图案重绘，产品形状改变，黑帧，水印。

**模型参数**（预留）：

```json
{
  "aspect_ratio": "9:16",
  "duration": 4,
  "seed": null,
  "motion_intensity": "轻微",
  "text_lock_enabled": true
}
```

### 4.4 提示词版本管理

- 每次修改模板中的提示词字段，`prompt_version` 递增
- 已生成的历史节点保留生成时的 `prompt_version`，便于追溯
- 重跑节点时使用**最新版本的提示词**，不沿用旧版本

## 5. 无限画布交互模式

### 5.1 核心原则

无限画布不是业务数据源，而是业务链路的可视化表达层。

- 画布坐标仅用于 UI 渲染，不参与业务逻辑
- 业务逻辑依赖 `product_type`、`asset_role`、`shot_key`、`instance_id`、`batch_id`
- 画布操作（拖拽/缩放/框选）是对业务状态的**可视化操作**，不是对业务数据的直接操作

### 5.2 模式一：模板设计模式

```
用途：创建和编辑分镜模板
入口：顶部工具栏 "模板设计" 切换按钮

画布内容：
  ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
  │S01   │──▶│S02   │──▶│S03   │──▶│S04   │──▶│S05   │──▶│S06   │
  │主视觉 │   │细节1 │   │细节2 │   │运镜  │   │场景  │   │尾帧  │
  └──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘

操作：
  - 点击节点 → 右侧抽屉编辑参数（时长、运镜、提示词、模型）
  - 拖拽节点 → 仅改变画布位置（不影响模板结构）
  - 连线 → 定义节点顺序
  - "保存模板" → 仅保存结构+参数，不保存素材

视觉：
  - 节点边框色：紫色（可变镜头）/ 珊瑚色（固定镜头 S06）
  - 节点内显示：shot_key + role_name + 时长
```

### 5.3 模式二：批量生产模式

```
用途：批量生成多条产品链
入口：上传素材包 → 选择模板 → "批量生成"

画布内容（N 条产品链，横向或纵向排列）：
  链1：S01→S02→S03→S04→S05→S06→MERGE
  链2：S01→S02→S03→S04→S05→S06→MERGE
  ...
  链N：S01→S02→S03→S04→S05→S06→MERGE

状态显示：
  ┌──────┐   灰色边框 = pending
  │ S01  │   蓝色跑马灯 = generating
  │ 主图 │   绿色边框 = success
  └──────┘   红色边框 = failed

批次看板（画布右下角浮窗）：
  批量生成任务 (batch_xxx)
  ████████████░░░░ 75%
  总计: 20  成功: 15  失败: 5
  ┌─────────────────────────────┐
  │ SKU2027-A01  ✅ completed   │
  │ SKU2027-A02  ✅ completed   │
  │ SKU2027-A03  ❌ failed      │ ← 点击定位
  │ ...                         │
  └─────────────────────────────┘

操作：
  - 点击批次看板中的失败项 → 画布自动定位到该链
  - 双击节点 → 预览生成结果（视频播放）
  - 右键节点 → 重跑 / 查看日志 / 查看审核记录
```

### 5.4 模式三：审核修正模式

```
用途：审核生成结果、修正失败节点
入口：点击任意生成完成的节点

审核面板（右侧抽屉）：
  ┌─────────────────────────────┐
  │ 节点: S01_main / SKU2027-A03│
  │ 状态: success               │
  ├─────────────────────────────┤
  │ 输入素材: [缩略图]          │
  │ 生成视频: [视频播放器]      │
  │ 时长: 4.1s (请求 4s)        │
  │ 模型: kling-v1.5            │
  │ 提示词: "产品展示视频..."   │
  ├─────────────────────────────┤
  │ 审核项:                     │
  │ ☑ 时长合格                  │
  │ ☑ 产品形状正常              │
  │ ☑ 文字可读                  │
  │ ☑ 无明显变形                │
  │ ☑ 无黑帧                    │
  │ ☑ 无水印                    │
  ├─────────────────────────────┤
  │ [通过]  [驳回]  [重跑]     │
  └─────────────────────────────┘

操作：
  - 审核通过 → review_status = approved
  - 审核驳回 → review_status = rejected，可修改提示词重跑
  - 重跑 → generation_job 新建，原记录保留
  - 查看合成视频 → 整链所有节点 approved 后自动合成
```

### 5.5 画布布局规则

- **模板设计模式**：6 个节点水平排列（x 间距 250px）

## 7. 质量审核体系

### 7.1 审核数据结构（在 ReviewRecord 中）

| 字段 | 类型 | 说明 |
|------|------|------|
| video_duration_ok | bool | 实际时长在请求时长的 ±0.5s 范围内 |
| product_shape_ok | bool | 产品主体形状无异常变形 |
| text_readable_ok | bool | 产品上的文字/日期清晰可读 |
| no_obvious_distortion | bool | 无明显的模型生成 artifact |
| no_black_frame | bool | 视频首尾无黑帧/花屏 |
| no_watermark | bool | 无模型厂商水印（业务要求） |
| review_status | enum | pending / approved / rejected |

### 7.2 审核流程

```
节点生成成功 (status=success)
  → 自动创建 ReviewRecord (status=pending)
  → 人工在审核面板中预览 + 勾选审核项
  → approved: 节点可进入合成
  → rejected: 节点标记为失败，可修改提示词/替换素材后重跑
```

### 7.3 MVP-3 不做自动质检

- 不做 AI 自动视觉检测（如用 VLM 判断产品变形）
- 不做自动清晰度评分
- 审核由人工完成，数据结构预留自动检测字段
- 后续可接入简单规则检测（如时长检测、黑帧检测）作为辅助

## 11. 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 真实模型 API 不稳定 | 生成失败率高，批量效率低 | 内置重试机制（最多 3 次），记录详细错误日志 |
| 模型生成质量不可控 | 产品变形/文字乱码，审核负担重 | 提示词约束 + text_lock 保护 + 预留自动检测字段 |
| 批量生成耗时过长 | 50 条链 × 6 节点 × 30s = 2.5 小时 | 支持并发调用（受限于模型 API 的并发限制），异步处理 |
| 模型成本不可控 | 批量生产成本超预期 | 记录每次调用 cost_estimate，提供成本预估 |
| SQLite 并发瓶颈 | 批量生成时数据库锁竞争 | 当前 MVP 阶段可接受，V1 考虑迁移到 PostgreSQL |
| instanceId 单实例模型的局限 | 多链操作时状态混乱 | 通过 resolveInstanceIdFromNode 缓解，V1 重新设计 |
